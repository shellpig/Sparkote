# 測試指南

> 測試流程、自動化測試與手動驗收清單。
> **撰寫時機**:各子階段測試項目於**規格定稿時即寫入**(與規格同期,acceptance-test 先行),供實作時據以建測試與自我驗收——不延到驗證才補。
> **角色歸屬**:測試項目於設計定稿時寫入;之後由 verifier 維護與執行。實作者照清單建測試、不得放寬清單,不改本檔(只列建議)。
> **範圍**:涵蓋**自動化測試(單元 / 整合,headless)**與**手動驗收(操作層)**兩類,見 §3 分層。

---

## 1. 本檔範圍與邊界

- **規格書「驗收」= 意圖層**:什麼狀態必須為真(what must be true),不寫怎麼測。
- **開發設計方針 = 接線層**:檔案 / 簽名 / 資料契約。
- **本檔 = 測試層**:用什麼測(自動化斷言 / 手動操作步驟)、跑哪個腳本、按哪顆鈕、看什麼。
- 同一條斷言不抄兩份;本檔每個清單項 link 指回規格書對應「驗收」段與設計方針對應契約。
- 規格定稿時若引用尚未定的 content(如事件文字),以中性描述先寫(如「顯示其內容頁」),實作時填實際字樣。

## 2. Godot 測試環境

引擎與工具路徑沿用 AfterTheModel 慣例(共用同一台 Godot 4.6.3)。

### 2.1 Headless 驗證命令

> 已知限制:sandbox 內直接跑會因無法開 `user://logs/godot*.log` 而 crash(signal 11);用 escalated 權限執行。

```
# 跑全部 GUT 測試
C:\_work\Godot_v4.6.3\Godot_v4.6.3-stable_win64_console.exe --headless -s addons/gut/gut_cmdln.gd -gdir=res://tests/unit -gexit
# 跑單一測試檔
C:\_work\Godot_v4.6.3\Godot_v4.6.3-stable_win64_console.exe --headless -s addons/gut/gut_cmdln.gd -gtest=res://tests/unit/test_game_state.gd -gexit
```

### 2.2 測試框架與檔結構

- **框架:GUT(Godot Unit Test)**,放 `addons/gut/`;只在 dev / 編輯器用,不進 release build。
- 測試檔放 `tests/unit/`,命名 `test_<system>.gd`,繼承 `GutTest`,以 GUT assert 斷言;GUT 自動探索並彙整 PASS / FAIL。
- `tests/fixtures/` — Phase 1 假資料(形狀正確、語意垃圾);自動化測試的內容來源。

| 測試檔(暫名) | 涵蓋 |
|---|---|
| `test_game_state.gd` | GameState 原子操作 + 序列化 round-trip |
| `test_config.gd` | Config 讀取 |
| `test_content.gd` | Content 載入 fixture + 查詢 + 壞資料報錯 |
| `test_requirements.gd` | requirements 判定 |
| `test_save.gd` | SaveService 存讀 |
| `test_ad_stub.gd` | AdService stub 成功 / 失敗 |
| `test_ui_navigation.gd` | UINavigation 根頁互斥 + overlay 還原 |
| `test_task_system.gd` | 1-B TaskSystem + DayCycle + 能量 |
| `test_exploration.gd` | 1-C 翻格 + requirements 阻擋 |
| `test_event_diary.gd` | 1-D 事件播放 + 日記資料 |
| `test_home_resource.gd` | 1-E 修復 + 資源點 |
| `test_phase1_smoke.gd` | Phase 1 端到端煙霧(§5) |

> 測試項目逐條對應規格書「驗收」與設計方針契約;新測試照清單建立,不放寬清單。

### 2.3 測試用 fixture 約定

- fixture 放 `tests/fixtures/`,不混進 `data/`。
- 至少含:數個任務(普通 / 高級)、1~2 張帶各類型格的小地圖(含一個帶 requirements 的阻擋格、一個出口格)、1 個含選項與獎勵的事件、數個物品 metadata、簡單家園(數個插槽 + 修復項)、一份 `config.json` 測試值。
- fixture 設計成可重現(固定內容),讓自動化斷言穩定。

## 3. 測試分層(自動 vs 手動,各擺什麼)

| 類別 | 對象 | 怎麼測 | 在 Phase 1 的比重 |
|---|---|---|---|
| **單元 / 整合(自動化,headless)** | 系統邏輯:狀態變更、能量規則、requirements、翻格結算、事件發獎 / 效果、存檔 round-trip、UINavigation 狀態機 | `test_*.gd` 斷言 + runner | **主力**(Phase 1 是純系統 + fixture,絕大多數可自動驗) |
| **手動驗收(操作層)** | 佔位面板的互動:點任務 / 翻格 / 開事件 overlay / 寫日記 / 修復,以及視覺回饋 / 提示語氣 | 開 debug 場景,照步驟點、看畫面 | 較少(只蓋 UI 互動與感受;1-A 無 UI,純自動) |

原則:**能用斷言驗的,一律進自動化清單**(回歸便宜);**需要人眼判斷畫面 / 手感 / 語氣的**,才進手動清單。1-A 全自動、無手動;1-B 起每個子階段兩類都有。

## 4. Phase 1 測試

> 各子階段驗收意圖見 `遊戲規格書.md > §20 Phase 1` 與對應系統段;接線見 `開發設計方針.md > §6 / §7`。

### 4.1 1-A 架構地基(全自動,無手動)

對應:規格書 §4~§18 驗收;設計方針 §6。

**GameState(`test_game_state.gd`)**
- [ ] `new_game` 後初始狀態正確(能量 0、資源空、未解鎖、旗標空、當日候選未骰)。
- [ ] 每個原子操作正確改值並發對應 signal(逐一:能量 / 資源 / 任務狀態 / 翻開 / 解鎖地圖 / 資源點採集記錄 / 旗標 / 事件完成 / 家園 / 日記)。
- [ ] `add_energy` 在上限內 clamp;達上限發 `energy_overflow`、超出不累積。
- [ ] `spend_energy` 不足回 false 且不扣;能量不會變負、不會超上限。
- [ ] `to_dict`→`from_dict` 還原後狀態與原本逐欄位一致(含當日任務、日記、旗標、已完成事件含選項)。

**Config(`test_config.gd`)**
- [ ] 以測試 path 讀 `tests/fixtures/config.json` 成功;型別化讀取(任務能量 by 類別、上限 / 門檻、槽位 / 候選、重採成本、每日次數、週超級門檻)值正確。

**Content(`test_content.gd`)**
- [ ] 載入 fixture 成功;查詢任務池 / 地圖 / 事件 / 物品 / 家園回正確物件。
- [ ] 壞資料(缺必填欄位、參照不存在的 id)→ 明確報錯,不靜默、不崩。
- [ ] (可選)鄰接雙向一致性檢查。

**Requirements(`test_requirements.gd`)**
- [ ] 五種條件(持有道具 / 完成事件 / 解鎖地圖 / 家園狀態 / 特產≥n)各自滿足與不滿足時回正確 bool。
- [ ] 多條件並存(AND)正確;`unmet_requirements` 列出正確的未滿足項。
- [ ] 判定即時對當前狀態計算(改狀態後重判結果跟著變)。

**SaveService(`test_save.gd`)**
- [ ] 存檔後讀回 = 原狀態(逐欄位)。
- [ ] 檔不存在 → 走 new_game、不崩;檔損毀 → 回報並 new_game、不崩。

**AdService stub(`test_ad_stub.gd`)**
- [ ] 請求 rewarded(stub)→ 觸發成功 signal。
- [ ] debug 失敗開關 → 觸發失敗 signal 並帶原因。

**UINavigation(`test_ui_navigation.gd`)**
- [ ] 任一時刻只一個根頁 active;`navigate` 切換正確。
- [ ] `open_overlay` 後底層輸入被凍結;`close_overlay` 還原到 caller 狀態 / 焦點。
- [ ] 疊兩層(事件 → 確認對話)後逐層關閉,各自正確還原。
- [ ] toast / 紅點不進 overlay stack、不凍結底層。

### 4.2 1-B 任務 + 能量

對應:規格書 §6 / §7;設計方針 §7.1。

**自動化(`test_task_system.gd`)**
- [ ] `roll_today` 產生 6 普通 / 3 高級候選;常用任務替代對應候選位;常用任務不增加可完成數。
- [ ] 選擇受槽位上限(2 普 / 1 高);超選被拒;完成前可改選。
- [ ] `complete_task`:完成「已選且未完成」者給正確固定能量一次;**只給固定能量、不發任何新手贈點**。
- [ ] 不能完成未選 / 已完成 / 超過槽位的任務。
- [ ] 看廣告額外高級(stub 成功)→ 從未選高級候選解 1 個;每日一次(再呼叫被拒)。
- [ ] 超級任務:有次數才能消耗,完成 +固定能量,每日最多 1;無次數時 no-op。
- [ ] `advance_day`:重置當日選擇 / 每日旗標、重骰候選、**保留結轉能量**。
- [ ] 週進度:當日所有已選任務完成累計天數;達門檻 → `super_task_credits +1` 並歸零計數。
- [ ] 當日全完成時發 `all_today_tasks_done`(供 1-D)。

**手動(佔位任務頁)**
- [ ] 開任務頁顯示能量、6 普 / 3 高候選、可選擇、完成鈕。
- [ ] 完成任務 → 能量顯示即時更新。
- [ ] 能量 >輕提示門檻 → 出現輕提示;近上限 → 提示更明確;**語氣溫和**(人眼確認文字不施壓)。
- [ ] 看廣告額外高級 debug 鈕、超級任務鈕(有次數時)、跳日鈕可操作。

### 4.3 1-C 探索翻格

對應:規格書 §8 / §9;設計方針 §7.2。

**自動化(`test_exploration.gd`)**
- [ ] `can_flip` 三關判定正確(相鄰 / requirements / 能量),擋下回正確原因。
- [ ] 只有「鄰接已開格的未知格」可翻;非鄰接者判定為不可翻。
- [ ] `flip_tile` 精準扣該格自帶成本、標記翻開;依類型結算:普通發現發獎、資源點首採加資源、裝飾加家具、出口解鎖指定圖、事件回 event_id。
- [ ] 被擋格(requirements 未滿足)無論能量多少都不能翻;在他處滿足條件後變可翻,翻時仍扣能量。
- [ ] 岔路:多選項並存可擇一;略過的格之後仍可補翻(無永久錯過)。
- [ ] 出口翻開後由 Exploration 讀 Content 目標圖起點並解鎖新圖,且新圖單一起點格自動翻開(新圖有可翻邊界、非死圖);可切換目前檢視地圖。
- [ ] 能量不足時翻格被擋、不扣、不變負。
- [ ] `flip_tile` 寫入日記(翻格數);事件解鎖紀錄只在 `EventSystem.resolve` 寫入一次。

**手動(佔位地圖頁)**
- [ ] 節點+連線板:格畫成形狀、相鄰格有連線(岔路可見);起點實心已開、其餘霧不畫。
- [ ] 可翻格顯示 `? + 成本`;點擊 → 扣成本 → 翻開顯示類型 → 鄰居浮現為新邊界 → 呈現 outcome(資源 inline / 事件交 overlay / 解鎖新圖提示)。
- [ ] 阻擋格顯示 🔒、不可點;條件滿足後才變可翻。
- [ ] 地圖切換器可在已解鎖地圖間切。
- [ ] **產 + 耗整條手感(fixture)**:領能量 → 翻幾格 → 能量見底 → 翻格被擋並有明確回饋。

### 4.4 1-D 事件 + 日記

對應:規格書 §10 / §14;設計方針 §7.3。

**自動化(`test_event_diary.gd`)**
- [ ] `play` 讀正確事件;`resolve` 正確發 rewards、套 effects、`mark_event_completed`(含選項)、寫日記。
- [ ] 有選項時依選擇正確分支(影響獎勵 / 日記文字)。
- [ ] `replay`:不重發獎勵、回傳當時所選選項。
- [ ] 日記資料由 `complete_task` / `flip_tile` / `resolve` 邊做邊記正確累積(任務 / 能量 / 翻格 / 事件),且事件不重複記錄。

**手動(事件 overlay + 日記頁)**
- [ ] 翻事件格 → overlay 依序播內容頁 → 末頁選項 → 選擇 → 結束**還原到原本那張地圖那一格**(caller restore)。
- [ ] overlay 期間底層地圖不被誤操作。
- [ ] 日記頁顯示逐日摘要;點已解鎖事件可回看(顯示當時選項、不重發獎)。
- [ ] 當日所有任務完成 → 跳心情筆記提示;選稍後仍可從日記補寫 / 編輯;寫筆記不動任何進度。

### 4.5 1-E 家園 + 資源點

對應:規格書 §11 / §12;設計方針 §7.4。

**自動化(`test_home_resource.gd`)**
- [ ] `repair`:持有指定資源時精準扣資源、修復等級 +1、寫日記(家園變化);資源不足回 false 且不扣。
- [ ] `place_furniture`:插槽只接受允許類型;放置 / 替換 / 收納正確;**完全不改能量 / 探索數值**。
- [ ] 資源點 `collect`:依每日次數可重採;重採扣 `recollect_cost` 能量、發 Content 產出、標記;次數用完當日不可再採,`advance_day` 後恢復。
- [ ] `get_collectable_points` 正確列出所有已解鎖地圖上已開的資源點 + 今日狀態。

**手動(佔位家園頁 + 集中採集頁)**
- [ ] 家園頁:修復項列表(需求資源 + 修復鈕 + 佔位等級)、插槽(佔位)、資源顯示;修復後佔位視覺變化。
- [ ] 集中採集頁:列出已開資源點 + 今日狀態 + **單點採集鈕**(無一鍵多採);各點顯示全域重採成本。

## 5. Phase 1 整合煙霧測試

對應:規格書 §2 核心循環;以 fixture 驗整條串接,不驗手感(手感在 Phase 2)。

**自動化(`test_phase1_smoke.gd`)**
- [ ] 端到端不崩:new_game → roll_today → 完成任務得能量 → 翻數格(含一個事件格、一個資源點、一個出口)→ resolve 事件發獎 → 修復家園一次 → 採集資源點 → `advance_day` → 候選重骰 / 能量結轉 → 存檔 → 讀檔 → 狀態逐欄位一致。
- [ ] 全鏈過程能量帳目正確(產出總額 − 消耗總額 = 結餘),無負值、無超上限。
- [ ] 日記在全鏈後含正確的當日摘要與事件紀錄。

**手動(debug 場景全鏈)**
- [ ] 從 debug 場景手動跑完上述一日:選任務→完成→翻格→事件→修復→採集→跳日,各面板顯示一致、overlay 還原正常。
