## 4. Autoload 一覽

| Autoload | 檔案 | 職責 | 依賴 |
|---|---|---|---|

## 5. 資料契約共識(跨子階段通用)

- **狀態欄位定義**:見規格書 §4.3,本檔不複製。`to_dict` 序列化的就是那整包。
- **存檔格式** = `GameState.to_dict()` 的輸出;版本欄位預留(後續遷移用)。
- **Content schema**:見規格書 §15;loader 對壞資料報錯不崩(缺必填欄位 / 參照不存在的 id → 明確錯誤,不靜默)。
- **相鄰**:格子明列鄰居 id(graph),與視覺座標分離;loader 可選驗證鄰接雙向一致。
- **數字歸屬**:逐格成本在格子資料、單點劇情給予在事件資料、全域節奏數字在 `config.json`(規格書 §5/§15)。

### 5.1 Phase 1 最小 Content JSON schema

Phase 1 fixture 與 Phase 2 真內容沿用同一份 key 形狀;fixture 只少內容量,不換 schema。

**`config.json`**:必填 `task_energy.normal` / `task_energy.advanced` / `task_energy.super`、`energy_cap`、`energy_hint_threshold`、`energy_near_cap_threshold`、`daily_slots.normal` / `daily_slots.advanced`、`daily_candidates.normal` / `daily_candidates.advanced`、`resource_point_recollect_cost`、`resource_point_daily_limit`、`weekly_super_threshold`。

**`tasks.json`**:必填 `normal`、`advanced` 兩個 Array;每筆任務必填 `id`、`text`。任務 id 全域唯一。

**`items.json`**:必填 `items` Array;每筆必填 `id`、`kind`、`display_name`。`kind` 允許 `resource` / `task_item` / `furniture` / `skin` / `collection`;家具另填 `furniture_type`;資源另填 `resource_tier`。

**`home.json`**:必填 `slots`、`repairs` Array。slot 必填 `id`、`position`、`accepts`(家具類型 Array);repair 必填 `id`、`max_level`、`levels`。每個 level 必填 `level`、`cost`、`visual_state`。

**`maps/<map_id>.json`**:必填 `id`、`name`、`world`、`region`、`resource_bias`、`start_tile`、`tiles`。tile 必填 `id`、`position`、`neighbors`、`cost`、`type`;可填 `requirements`。各 `type` 額外必填:
- `start`:無額外欄位。
- `discovery`:至少一個 `rewards` 或 `text`。
- `resource_point`:必填 `first_rewards`、`collect_rewards`。
- `event`:必填 `event_id`。
- `decor`:必填 `reward_id`。
- `exit`:必填 `target_maps` Array。

**`events/<group>.json`**:必填 `events` Array;每筆事件必填 `id`、`title`、`pages`、`replayable`。可填 `choices`、`rewards`、`effects`、`diary`。

**參照檢查**:loader 至少檢查 map `start_tile` 存在、tile neighbors 存在、`event_id` 存在、`target_maps` 存在、獎勵 / 消耗 / 家具 / 物品 id 存在、home slot `accepts` 對得上家具類型。

## 6. 1-A 架構地基(實作契約)

> 機制與驗收意圖見 `遊戲規格書.md > §4~§18 與 §20 Phase 1`。以下是 1-A 的實作接線。1-A 目標:所有架構件以「可被呼叫的契約 + 最小真實行為」立起來,headless 全綠,後續子階段有 seam 可接,**不含真內容、不含正式面板**。

### 6.1 GameState — 原子 API

狀態欄位見規格書 §4.3。對外原子方法(以行為描述,不放程式碼):

| 方法 | 參數 | 回傳 | 行為 |
|---|---|---|---|
| `new_game` | — | — | 重置初始狀態(不在此 roll_today,由 DayCycle 接) |
| `to_dict` / `from_dict` | (dict) | dict / — | 序列化 / 還原全狀態 |
| `get_energy` / `can_afford` | (cost) | int / bool | 查詢 |
| `add_energy` | amount, source | 實際加值 | clamp 上限;達上限發 `energy_overflow` |
| `spend_energy` | cost | bool | 不足回 false 不扣;不會變負 |
| 任務(原子) | set_today_candidates、choose/unchoose 普通·高級、set_extra_advanced、mark_task_completed、is_task_completed、mark_super_task_done、save/remove_favorite | bool / — | 受槽位上限;不含完成編排(在 TaskSystem) |
| 探索(查詢) | is_tile_revealed、is_adjacent_to_revealed、mark_revealed、unlock_map、is_map_unlocked | bool / — | 純狀態;`unlock_map` 只記 map id,起點格由 Exploration 讀 Content 後呼叫 `mark_revealed` |
| requirements | meets_requirements(reqs)、unmet_requirements(reqs) | bool / Array | 即時對當前狀態判定 |
| 事件 | is_event_completed、get_event_choice、apply_rewards、apply_effects、mark_event_completed | … | 發獎 / 效果為原子;播放在 EventSystem |
| 資源 | get/add/spend_resource、get_point_collect_count、mark_point_collected | int / bool / — | 扣不足回 false;採集次數上限由 ResourcePoint 讀 Config 後判定 |
| 任務道具 | has/add/consume_task_item | bool / — | 含關鍵物 |
| 家園 | get/set_repair_level、place_furniture、remove_furniture | int / bool / — | 純狀態;插槽接受類型由 HomeSystem 讀 Content 後先檢 |
| 擁有物 | add/has_furniture、add/apply_skin、add_collection | — / bool | 背包 view 用 |
| 變現 | has_no_ads、is_purchased、grant_purchase、mark_daily_free/ad_supply/ad_extra | bool / — | |
| 旗標 | set_flag、get_flag、is_system_unlocked | — / Variant / bool | |
| 日記 | ensure_diary_day、log_to_diary、write_mood_note、get_diary | — / dict | |

**signals**:`energy_changed` / `energy_overflow` / `resources_changed` / `tile_revealed` / `map_unlocked` / `today_changed` / `event_completed` / `system_unlocked` / `home_changed` / `diary_updated` / `day_advanced` / `state_loaded`。

> 顆粒度:第一版採「整類刷新」signal(如 `resources_changed` 不帶 delta);若 UI 證實需要更細(如帶變動的資源 id),再加,不預先加。

### 6.2 Config(全域設定)

- 預設讀 `data/config.json`(規格書 §5 的值);測試可用明確 path 載入 `tests/fixtures/config.json`。
- 提供型別化讀取(任務能量 by 類別、能量上限 / 門檻、槽位 / 候選數、重採成本、每日採集次數、週超級門檻)。
- read-through:呼叫端每次讀最新值。

### 6.3 Content

- 載入並查詢:任務池、地圖(by id)、事件(by id)、物品 metadata、家園定義。
- Phase 1 讀 `tests/fixtures/`(極小假資料:1 圖 / 1 事件 / 數任務 / 數物品 / 簡單家園),只證 pipeline。
- 壞資料報錯不崩;參照完整性可選檢查(事件 id / 出口圖 id / 物品 id 存在)。

### 6.4 Requirements

- 判定器:輸入條件陣列,對當前 `GameState` 即時回 bool;另回未滿足項清單。
- 條件類型(第一版):持有任務道具 / 完成指定事件 / 解鎖指定地圖 / 達成指定家園狀態 / 擁有≥n 指定世界特產。
- **實作位置(已定)**:放 `GameState` 當查詢方法(`meets_requirements` / `unmet_requirements`),不獨立成系統——它純讀 GameState,獨立只會多一跳;§4 表已列為 GameState 方法。

### 6.5 SaveService

- 存:`GameState.to_dict()` → 本機檔(`user://`,JSON)。
- 讀:檔 → `GameState.from_dict()`;發 `state_loaded`。
- round-trip 無損;檔不存在 / 損毀時走 new_game 並回報,不崩。
- 雲端 / 換機:非 1-A 範圍(Phase 5)。

### 6.6 AdService(stub)

- 介面對齊 spike:請求 rewarded、成功 signal、失敗 signal(帶原因)。
- Phase 1 stub:預設呼叫即發成功 signal(不播);必須提供 debug fail mode 模擬失敗,以驗優雅降級路徑。
- rewarded 廣告單元 ID 等常數只出現在此一處(Phase 3 換真值)。

### 6.7 EventSystem(僅 seam)

- 1-A 只立可被呼叫的接縫:`play(event_id)` → 經 UINavigation 開 overlay 顯示佔位;`resolve(event_id, choice)` → 呼叫 GameState 發獎 / 效果 / 標記完成 / 寫日記 → 關 overlay 還原 caller。
- 真正播 pages / choices / 插圖 → 1-D。

### 6.8 UINavigation(契約 + 薄實作)

- 根頁集合(列舉):任務 / 地圖 / 家園 / 背包 / 日記 / 商店;同時只一個 active。
- `navigate(page)`:切換 active 根頁。
- `open_overlay(kind)` / `close_overlay()`:overlay stack(事件 / 商品詳情 / 確認對話);開時凍結底層輸入,關時還原 caller 狀態 / 焦點。
- `toast` / 紅點:非阻擋,獨立於 stack。
- 1-A 薄實作:純色佔位切換,無轉場 / skin(Phase 4 加表現)。

### 6.9 啟動 / debug 場景

- `scenes/main.tscn`:啟動使 autoload 就緒;提供最小 debug 入口(可觸發 headless 測試所驗的路徑)。
- 正式佔位面板自 1-B 起長;1-A 不做面板。

### 6.10 1-A 待確認(剩餘)

已定案:`Config` 命名、邏輯系統一律 autoload(§4)、Requirements 放 GameState(§6.4)、`advance_day` 編排放 `DayCycle`(GameState 只提供原子 day+1 與重置 helper,不依賴 TaskSystem)。

仍待開工時定:
- 存檔檔名 / 版本欄位格式。

## 7. 1-B ~ 1-E(實作契約)

> 各子階段機制與驗收意圖見 `遊戲規格書.md > §20 Phase 1` 對應列與系統段;本段是接線契約。所有編排方法呼叫 §6.1 的 GameState 原子 API + 讀 Content / Config,不繞過 GameState 改狀態。佔位面板自 1-B 起經 `UINavigation` 掛為根頁,純色 + 文字,無 skin。

### 7.1 1-B 任務 + 能量

新增系統:`DayCycle`、`TaskSystem`。第一個佔位根頁:任務頁。

**`DayCycle`**

| 方法 | 行為 |
|---|---|
| `advance_day` | 經 GameState 原子:day+1(發 `day_advanced`)→ 重置當日任務狀態 + 每日領取 / 廣告旗標 → 週檢查(本週全完成天數達門檻 → `super_task_credits +1` 並歸零計數)→ 呼叫 `TaskSystem.roll_today` |

> GameState 提供原子 helper(increment_day / reset_today / reset_daily_flags / add_super_credit);`advance_day` 的編排放 `DayCycle`,使 GameState 不依賴 TaskSystem(§6.10 決策)。

**`TaskSystem`**

| 方法 | 參數 | 回傳 | 行為 |
|---|---|---|---|
| `roll_today` | — | — | 從 Content 任務池抽候選(普通6/高級3)→ 用 GameState 常用任務替代對應候選位 → `GameState.set_today_candidates` |
| `complete_task` | task_id | — | 驗證已選且未完成 → `GameState.add_energy(Config.task_energy[類別], "task")` → `GameState.mark_task_completed` → `log_to_diary`(完成任務 / 獲得能量)→ 若當日所有已選任務皆完成且今日尚未計入 → 累計本週全完成天數、發 `all_today_tasks_done`(供 1-D 心情筆記)→ emit `TaskSystem.task_completed`。**只給固定能量,不發任何新手贈點。** |
| `unlock_extra_advanced_via_ad` | — | — | 呼叫 `AdService.request_rewarded`;成功 callback → 從未選高級候選挑 1 個 → `GameState.set_extra_advanced` → `GameState.mark_daily_ad_extra`;失敗 → 回饋不解鎖 |
| `complete_super_task` | text | bool | 有 `super_task_credits` 才行 → 扣 1 → `add_energy(super)` → `GameState.mark_super_task_done`;每日最多 1 |

新增 signals:`TaskSystem.task_completed` / `TaskSystem.all_today_tasks_done`。

**佔位任務頁**:顯示能量、普通6/高級3 候選、選擇(受槽位)、完成鈕、看廣告額外高級鈕(debug)、超級任務鈕(有次數時)、debug 跳日鈕;綁 GameState / TaskSystem signals 刷新。

**1-B 待確認**:週邊界處理(rolling 7 日 vs 自然週);候選抽取是否去重 / 用種子(可重現測試);常用任務替代候選的對位規則(第 n 個常用 → 第 n 個候選)。

### 7.2 1-C 探索翻格

新增系統:`Exploration`。到此產 + 耗整條通。佔位根頁:地圖頁。

**`Exploration`**

| 方法 | 參數 | 回傳 | 行為 |
|---|---|---|---|
| `can_flip` | map_id, tile_id | {ok, reason} | 三關:`is_adjacent_to_revealed` + `GameState.meets_requirements(tile.reqs)` + `can_afford(tile.cost)`;擋下回原因(too_far / blocked / no_energy) |
| `flip_tile` | map_id, tile_id | outcome | 檢 `can_flip` → `spend_energy(tile.cost)` → `mark_revealed` → `log_to_diary`(翻格數)→ 依類型結算:普通發現 `apply_rewards`;資源點**首採** `add_resource`(量取自格資料);裝飾 `add_furniture`;出口讀 Content 目標圖起點後 `unlock_map` + `mark_revealed`(可多張)；事件 → 回 `{event_id}` 交 `EventSystem.play`(事件日記由 resolve 寫)→ 回 outcome |

> 解鎖新圖時由 `Exploration` 讀 `Content.get_map(map_id).start_tile`,再呼叫 `GameState.unlock_map(map_id)` + `GameState.mark_revealed(map_id, start_tile_id)`(發 `map_unlocked` + `tile_revealed`),確保新圖有可翻邊界。第一版單一起點,不做多入口。
| `set_current_map` / `get_current_map` | map_id | — / id | 多地圖切換:在 `GameState.unlocked_maps` 間切目前檢視圖 |
| `get_map_view` | map_id | dict | 給地圖頁渲染:已開格 + 可翻候選格 + 各格類型 / 成本 / 阻擋狀態 |

**佔位地圖頁(節點 + 連線板)**:每格畫成簡單形狀(置於格資料的視覺座標),格與格之間依相鄰 graph 畫**連線**(呈現岔路)。狀態以簡單圖示區分:
- **已翻開**:實心 + 類型字(起點 / 資源 / 事件 / 出口 / 裝飾)。
- **可翻**(鄰接已開格):`? + 成本數字`,點擊 → 扣成本 → 翻開 → 鄰居浮現為新邊界 → 呈現 outcome(資源 inline / 事件交 overlay / 解鎖新圖提示)。
- **阻擋**(requirements 未滿足):🔒,不可點,顯示被什麼擋。
- **霧**(尚未鄰接任何已開格):不畫出,鄰居開了才浮現。
另含地圖切換器(已解鎖地圖間切)。Phase 1 上視角佔位,Phase 4 換等角美術(座標 / graph 同一份)。

**1-C 待確認**:節點板的佔位繪製細節(形狀 / 連線樣式);切換地圖的入口(地圖頁內 vs 導航)。
